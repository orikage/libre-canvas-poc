## 1. プロジェクト発端・ユーザーの声

### 元の要求

> クリスタのOSS版を作りたい。  
> ブラシとかの互換性を保ったまま、軽量版としてマルチプラットフォーム対応にしたい。  
> その上、訴訟対策も入れといて欲しい。  
> ユーザ目線と運営目線を持って、企画書を作成して。  
> その際、具体的にどのようなものが必要か、それがどう言う風に行えば最適解かを考えて。  
> **1ユーザ目線の意見としては、ちょろっと書きたいだけなのにサブスクで払いたくない + PCとiPadとかでデータを同期しておきたいみたいな感じ。**

## 2. 企画書（初版）

### タイトル

**「次世代オープンソース・ペイントツール "LibreCanvas"（仮称）」**

### 1. プロジェクトの背景と目的

高機能化とサブスクリプション化が進むペイントソフト市場において、「たまにしか描かないのに固定費を払い続けたくない」「重すぎて起動がおっくう」というユーザーのペインを解消する。

目指すのは、**軽量・買い切り不要（完全無料）・複数デバイス間でのシームレスな同期**を実現しつつ、既存のクリエイティブ資産（ブラシ等）を無駄にしないOSSペイントツール。

### 2. ユーザー目線・運営目線のコアバリュー

#### ユーザー目線（ライト層・ホビー層）

- **「描きたい時にすぐ描ける」**: 重厚長大な起動プロセスを排除した圧倒的な軽量性
- **サブスクからの解放**: ずっと無料で使える。課金圧力が一切ないクリーンな環境
- **シームレスな体験**: PC（Windows/Mac/Linux）で描きかけのラフを、出先やリビングでiPadからそのまま続きを描ける同期機能
- **完全なノイズレス**: 広告・トラッキング・ポップアップ一切なし

#### 運営目線（OSSメンテナー・開発コミュニティ）

- **法的リスクの完全排除**: 既存の商用ツール（特にCELSYS社など）からの訴訟・警告を回避するクリーンルーム設計と機能のモジュール化
- **インフラコストの最小化**: 中央集権的なサーバーに依存せず、ユーザー側で同期インフラを用意できる「ローカルファースト」な設計により、運営のサーバー維持費をゼロに近づける
- **持続可能な開発モデル**: GitHub Sponsors等を通じたドネーションモデルとコミュニティ主導のエコシステム形成

### 3. 具体的な要件と「最適解」のアーキテクチャ

#### ① マルチプラットフォームと圧倒的な軽量化

**最適解の技術スタック: Tauri + Rust + WebGPU (または Flutter)**

Electronのようなメモリ消費が激しい技術を避け、Rustによる高速なバックエンド処理とWebフロントエンド技術を組み合わせることで、PCでもタブレット環境でもネイティブアプリに近いパフォーマンスを軽量に実現する。

#### ② ブラシ等の互換性の保持

- **独自標準形式 (.lcb - LibreCanvas Brush)**: JSONやYAMLベースの人間が読めるオープンな記述形式
  - 格納内容: 形状画像（PNG）、テクスチャ、筆圧感度曲線、混色アルゴリズムのパラメータ
- **汎用描画エンジン**: あらゆるツールのパラメータを受け入れられる柔軟なシェーダーベースエンジン
- **動的プラグイン化**: ブラシ読み込み機能をWASMモジュールとして動的ロード可能にする
- **コミュニティ主導**: 開発者が公式に `.sut` 読み込み機能を配布するのではなく、「ユーザーや有志が作ったインポーターを後から追加できる仕組み」だけを提供

**ユーザー体験フロー**:
1. ユーザーが `.abr` や `.sut` ファイルをD&D
2. 対応インポーターがあれば即座に `.lcb` 形式に変換してブラシパレットに追加
3. 不足時は「このファイル形式を開くには〇〇インポーターが必要です。コミュニティサイトで探しますか？」と表示

#### ③ 複数デバイス間のデータ同期

アプリ本体はネットワーク通信を一切行わない。ユーザーが普段使っているクラウドストレージ（OneDriveやiCloud）のフォルダに保存するだけで同期完了。

**各プラットフォームでの実装**:
- **PC (Win/Mac/Linux)**: アプリは指定されたローカルディレクトリ（例: `C:\Users\Name\OneDrive\LibreCanvas\`）のファイルを直接開き、上書き保存
- **iPad**: OS標準の「ファイルアプリ」経由でOneDrive等の外部ストレージにアクセス。iCloud Driveを使う場合はネイティブのDocument Provider APIを使用

**ローカルバックアップ（bk機能）**:
- 同期フォルダには常に最新ファイル（`artwork.lcv`）だけを上書き保存
- 各デバイスのローカルストレージ（同期されない一時フォルダ）に `artwork_20260411_1650.bak` 形式で履歴バックアップを自動生成
- クラウドの同期ミスでファイルが壊れても、各デバイスのローカルに無傷のバックアップが残る

### 4. 訴訟対策（リーガル・リスクマネジメント）

#### 絶対に避けるべきこと（ブラックリスト）

- `.sut` 等の独自フォーマットをリバースエンジニアリングしてコアに組み込む
- 「クリスタ互換」「クリスタの代替」を前面に出したマーケティング
- CelSYS社等の商標・ロゴの無断使用

#### 実践すべき対策（最適解）

| 対策 | 内容 |
|------|------|
| **クリーンルーム設計** | 既存商用ツールのソースコードを参照せず、仕様のみを公開情報から実装 |
| **機能のモジュール化** | 互換性機能はコアから分離し、コミュニティプラグインとして提供 |
| **オープンライセンス** | MIT または Apache 2.0 ライセンスを採用 |
| **コントリビューションガイド** | DCO（Developer Certificate of Origin）を導入 |
| **記録の保持** | 設計決定の根拠と情報源を記録し、独自開発であることを証明できる状態を維持 |

## 3. 競合分析・差別化戦略

### 既存無料アプリとの比較マトリクス

| 製品 | LibreCanvas | ibis Paint | MediBang | Krita |
|------|:-----------:|:----------:|:--------:|:-----:|
| 完全無料 | ✅ | △（広告あり） | △（広告あり） | ✅ |
| 広告なし | ✅ | ❌ | ❌ | ✅ |
| トラッキングなし | ✅ | ❌ | ❌ | ✅ |
| 軽量起動 | ✅ | ✅ | ✅ | ❌ |
| BYOS同期 | ✅ | ❌ | ❌ | ❌ |
| データ主権 | ✅ | ❌ | ❌ | ✅ |
| PC+iPad対応 | ✅（予定） | ✅ | ✅ | △ |
| OSS | ✅ | ❌ | ❌ | ✅ |

### 「企業が提供する無料」vs「OSSの無料」の決定的な違い

アイビスやメディバンは非常に優秀なツールだが、営利企業である以上、機能の提供は「自社エコシステムへの囲い込み」と「マネタイズ（広告収益・サブスクリプション）」が前提となる。

**ユーザーのペイン**: 「ちょっと描きたいだけ」の時にアプリを開くと、全画面広告が表示されたり、使いたいブラシがロックされていて「動画広告を見てロック解除」を要求されたりする。これは思考の分断を招く。

**運営側のコスト構造**: 企業側は全ユーザーのクラウド同期データを自社サーバーで抱え込むため、莫大なインフラコスト（AWS等のストレージ・通信費）が発生する。これを回収するために、さらなる広告表示やサブスクリプションへの誘導が避けられない構造になっている。

### LibreCanvasの圧倒的差別化ポイント

#### ① 「完全なノイズレス」による純粋な描画体験

広告SDK・ユーザー行動トラッキングコード・自社サービスへのログインポップアップを一切搭載しない。アプリのバイナリ自体が描画エンジンに特化しているため、アイビス等と比較してバックグラウンドでの通信やメモリ消費が劇的に少なくなる。バッテリー駆動のタブレットでも極めて長時間の作業が可能になる。

#### ② 「BYOS（Bring Your Own Storage）」による真のマルチデバイス同期

企業製アプリのクラウド同期は便利だが、「容量上限」と「サービス終了・規約変更のリスク」と常に隣り合わせ。

LibreCanvas自体は同期サーバーを提供しない。その代わり、標準プロトコルでの透過的な同期機能を実装する。

**最適解のユースケース**: ユーザーは自宅のNAS（Synology等）やOneDrive/iCloud Driveを同期先として指定。アプリはそのフォルダにファイルを保存するだけ。

#### ③ インフラの非中央集権化とGitベースバージョン管理

- アプリ内に同期用のサーバーを持たず、プロトコルのみを提供
- ユーザー自身が構築した自宅サーバー（Dockerコンテナ等）やNAS、S3互換ストレージを同期先として指定できる設計
- アートワークの差分をGitライクに管理できる機能を将来的に実装（GitHubへの直接バックアップ・履歴管理）

#### ④ アーキテクチャの軽量性（「描画」への特化）

Kritaのような既存の強力なOSSは、多機能化の代償としてQtやC++の重厚なフレームワークに依存しており、特にタブレット端末では動作が重くなりがちである。LibreCanvasはリソース効率を極限まで高める。

#### ⑤ リーガル・クリーンとエコシステムの分離

本体は「クリーンな描画ソフト」であり続け、互換性機能は「ユーザーの自由な拡張」という建付けにする。

### プロジェクトのポジショニング

> **「SAIのように軽くて線が美しい、iPadでも動くOSSを作ります」**

「多機能」ではなく「究極の描き味」を軸にポジションを確立することで、熱烈な支持（コントリビューター・寄付）を集める戦略。

### 市場動向（PoC当時の情報）

- CLIP STUDIO PAINTはサブスクリプションモデルへ移行し、既存ユーザーから強い反発が起きている
- Adobe製品もサブスクリプション化により「Adobe離れ」が加速
- ライト層の「取り残し」が発生しており、新規参入のチャンス

## 4. 新規事業 Gate 1 審査フレームワーク

### Gate 1 審査向け思考プロセス

#### Phase A: ターゲットと課題（Who / Problem）

- **Who**: 「ちょっと描きたいだけ」のライトユーザー・ホビー層
- **Problem 1**: 過剰な多機能化と重量化 — プロユース向けの機能追加に伴い起動時間・メモリが増大。「ちょっと描きたい」という初期衝動を阻害
- **Problem 2**: サブスクリプションによる経済的ロックイン — たまにしか使わないユーザーへの継続課金負担。解約すると過去の自作データの編集すら困難
- **Problem 3**: データ主権の喪失 — 複数デバイス間のデータ同期が各企業の独自クラウドに依存

#### Phase B: ソリューションと提供価値（What / Solution）

**プロダクト**: 超軽量・完全無料のOSSペイントツール「LibreCanvas（仮称）」

**バリュープロポジション**:
1. Rust / WebGPUベースの「描くこと」に特化した爆速の起動と軽快な動作
2. BYOS（Bring Your Own Storage）型同期 — WebDAV、S3互換API、Git連携をサポートし、ユーザー自身のサーバーや任意のクラウドでデータを同期・管理できる「データ主権」の回復
3. 商用アプリのブラシ等の資産は、外部の「独立したコンバーター」を通じて安全に変換して読み込む（本体側での法的リスク回避）

#### Phase C: 市場動向と外部環境（Market / Trend）

現在の市場は「高機能・高価格化」へ向かっており、ライト層の取り残しが発生している。これは新規参入の大きなチャンス。

#### Phase D: 競争優位性（Unfair Advantage）

- ゼロから設計するため「レガシーコードの呪い」がない
- 中央サーバーを持たないためインフラコストが本質的にゼロ
- OSSであるためコミュニティの貢献によって機能が拡張される

### Gate 1 評価結果：高く評価できる点

1. **ターゲットと課題の設定が明確** — 「たまにしか描かない人」「サブスク払いたくない人」という具体的なペルソナ
2. **提供価値とソリューション** — BYOSによる真のデータ主権という独自の切り口
3. **競争優位性と持続可能性** — レガシーコードなし、インフラコストゼロという構造的優位性
4. **法的・技術的実現性** — クリーンルーム設計とモジュール化による訴訟対策

### Gate 1で指摘される致命的な弱点（Blind Spots）

#### 致命的弱点①：開発の持続可能性（マネタイズの欠如）

GitHub Sponsorsは個人開発の生活を支えるほどの収入にはなりにくい。開発が止まった場合のコミュニティへのダメージをどう防ぐか？

**改善案**: 
- 個人プロジェクトとして割り切り、「趣味の範囲で持続する」という前提を明確にする
- 将来的にプラグインマーケットプレイスを作り、手数料収入を得る設計にする

#### 致命的弱点②：ターゲット層とUXの矛盾（BYOSの壁）

「ちょっと描きたいだけ」の**ライトユーザー**が、OneDriveのフォルダ設定やNASの構築を自分でやれるか？ これはターゲット層（技術に詳しくないホビー層）には難しすぎる。

**改善案（最終的に採用）**: 
- **デフォルト動作はシンプルに**: 「保存ボタンを押すだけ。ファイルはドキュメントフォルダに入る」
- BYOSはあくまで「上級ユーザー向けのオプション機能」として位置づける
- Onboardingで「あなたが使っているクラウドはどれですか？」と聞き、選んだクラウドのデフォルトフォルダを自動設定する

#### 致命的弱点③：スイッチングコストの過小評価

クリスタユーザーは何百本というブラシと何年分のデータを持っている。`.lcv` 形式への移行コストをユーザーが負担するのか？

**改善案**:
- フォーマット変換ツールを別途提供（これ自体は法的リスクが低い）
- 「既存のJPG/PNG/PSD画像を読み込んで上から描ける」機能を優先実装する

## 5. アーキテクチャ：同期・データ設計

### 最終的な設計思想

> **「同期はOS・クラウドに丸投げする」**

単なるファイル群として保存し、同期は既存のクラウドストレージ（OneDrive、iCloud、Dropbox、あるいは自前構築のNextcloudなど）のクライアントアプリに丸投げするアプローチ。

### なぜこのアプローチが「最強」なのか

| メリット | 詳細 |
|---------|------|
| **開発・維持コストが完全にゼロ** | 同期処理のバグ（通信エラー時のリトライ、認証トークン切れ等）に悩まされることが一生ない。同期が止まったら、それはMicrosoftやAppleの問題 |
| **ユーザーの自由度** | OneDriveを使いたい人はOneDriveフォルダへ、iCloudを使いたい人はiCloudフォルダへ保存するだけ。ユーザーが既に構築しているエコシステムにただ「乗っかる」だけ |
| **圧倒的な軽さ** | アプリ本体はネットワーク通信を一切行わないため、完全なオフラインアプリとして動作。動作の軽快さとセキュリティ（情報漏洩ゼロ）が担保される |

### Core-Shell（コアと殻）モデル

```
[The Core（共通描画エンジン）]
  ↓  Rust + WebGPU + WASM
  ↓  プラットフォーム非依存
  ↓
[The Native Shell（本命アプリ版）]
  ├── PC: Tauri（OSのファイルシステムと直接対話、自動バックアップ、完全ショートカット制御）
  └── iPad: 将来的に Tauri Mobile または SwiftUI ラッパー
```

「Webで最低限動くもの」をベースラインとして構築することで、アーキテクチャをクリーンに保ち、特定のプラットフォームに依存しないOSSとしての寿命を劇的に延ばす。その上で、Tauri等を用いた「ネイティブの殻」を被せて本命アプリとして配布する。

### データフロー

```
[ペイント操作]
     ↓
[LibreCanvas コア（通信なし・完全ローカル）]
     ↓ 保存
[ユーザーが指定したローカルフォルダ]
     ↓ OS/クラウドクライアントが自動同期
[OneDrive / iCloud Drive / Dropbox / NextCloud 等]
     ↓
[別デバイスのローカルフォルダ]
     ↓
[別デバイスのLibreCanvas]
```

### ローカルバックアップ（bkファイル）戦略

```
同期フォルダ: artwork.lcv       ← 常に最新ファイルのみ上書き
ローカルフォルダ: artwork_20260411_1650.bak  ← 各デバイスでローテーション自動生成（同期対象外）
```

**保存の詳細**:
- `journal.log` は各デバイスの `AppData` / `Application Support` 内にローカル保存
- クラウド側のファイルが破損しても、このログからストロークを再実行（Playback）することで絵を100%復元できる

### ファイル構造：分散型 `.lcv` 形式

単一の巨大なバイナリではなく、**「ディレクトリ構造のパッケージ」**を採用。

```
my_artwork.lcv/          ← ディレクトリ
├── meta.json            ← キャンバスサイズ、レイヤー情報等のメタデータ
├── tiles/
│   ├── layer_0/
│   │   ├── 0_0.bin     ← 256x256タイル（zstd圧縮済み）
│   │   ├── 0_1.bin
│   │   └── ...
│   └── layer_1/
│       └── ...
└── thumbnail.png        ← プレビュー用サムネイル
```

**同期最適化**: ユーザーが1筆描くごとに、そのストロークが影響した「特定のタイルのバイナリ（.bin）」だけが更新される。OneDriveやDropboxは「変更された小さなファイル」だけを差分同期するため、4Kの大容量ファイルでも数KBの通信で同期が完了する。

### 競合ファイルへの対処

**問題**: PCとiPadで同時に同じファイルを開いて編集してしまった場合、OneDrive側で「競合コピー」が発生する可能性がある。

**対処**: 「bkは各コンピュータに置く」設計が完璧にカバーする。
- 同期フォルダには最新ファイルのみ上書き
- ローカルバックアップが無傷で残るため、いつでも復旧可能
- 将来的にConflict Resolutionロジックを追加予定

## 7. ブラシ互換性と訴訟対策の詳細

### ブラシ互換性における「法的・技術的」課題

`.sut`（クリスタ）や `.abr`（Photoshop）はそれぞれの企業の著作物であり、これをリバースエンジニアリングしてコアに組み込むと訴訟リスクが生じる。

### 最適解：プラグイン型「ユニバーサル・インポーター」方式

#### ① 本体（Core）の仕様

- ネイティブ形式 `.lcb`（JSON/YAMLベース）のみを直接読み込む
- 外部プラグイン（WASM）をロードするための標準的なAPIを提供する
- 本体自体は「クリーンな描画ソフト」であり続ける

#### ② インポーター・レイヤー（訴訟対策の肝）

- `.sut` / `.abr` 等の読み込み機能は、**コミュニティが開発したWASMプラグイン**として提供
- 開発者（プロジェクトオーナー）は公式にリバースエンジニアリングコードを配布しない
- 「ユーザーが自己責任でインポーターを追加できる仕組み」だけを提供することで法的リスクを分散

### ブラシ互換の技術的アプローチ

**`.abr`（Photoshop）に関して**:
- 既存のOSS（KritaやGIMP）がこれを解析し、実用レベルで読み込めるようになっている
- 最新の「デュアルブラシ」や「3Dテクスチャ」等の高度な機能は完全再現できないが、基本的なブラシは移植可能

**`.sut`（クリスタ）に関して**:
- 公式の仕様書が存在しないため、コミュニティが非公式に解析したコードが必要
- コア本体への組み込みは行わず、外部プラグインとして実装

## 8. 描画エンジン：技術的深掘り

### なぜ既存企業は「軽い描画」ができないのか？

#### ① 20年分の「レガシーコード」という呪い

PhotoshopやクリスタはGPUレンダリングが一般的でなかった時代（CPUで計算していた時代）のC++コードがベース。最新のWebGPUやRustベースの完全な非同期アーキテクチャに「ごっそり」乗り換えようとすれば、過去20年で作ってきた何千という機能（古いフィルター、テキストエンジン、謎のプラグイン）がすべて動かなくなる。書き直すコストは企業にとって割に合わない。

#### ② 「機能肥大化」によるキャッシュの崩壊

「色調補正レイヤー」「レイヤースタイル（ドロップシャドウ等）」「ブレンド条件の複雑化」が存在するため、「レイヤー2の不透明度を1%変えたら、上に乗っている色調補正レイヤーの影響で、レイヤー50までの見た目が全部変わる」といった依存関係のスパゲッティが発生しており、簡単にキャッシュを保持できない構造になっている。

#### ③ 最大の計算コスト「ブラシ・ダブ（Dab）の生成」

線を引くという行為は、PCの内部では「1秒間に数千個のブラシのスタンプ（Dab）を連続してペタペタと押す」処理。現代のソフトは、このDab一つ一つに対して「筆圧、傾き、テクスチャのランダム回転、デュアルブラシの合成、水彩の境界エッジ計算」を同時に行う。

### なぜ LibreCanvas なら速度要件を満たせるのか？

- **機能を2015年水準に絞る**: キャッシュの依存関係スパゲッティを最初から作らない
- **RustによるSIMD**: 混色計算を1回の命令で4〜8ピクセル同時処理
- **タイルベース管理**: 変更されたタイルのみを更新対象とし、計算を局所化
- **Dirty Rect最適化**: ペン先の半径Rに補正の遊び分を加えた「最小の矩形」のみを計算対象にする

### 描画アーキテクチャ最終決定：CPU SIMD + GPU 表示専念

#### 検討経緯

1. **WebGPU Compute Shader案（却下）**: 混色の直列依存性とメンテナンス性の問題
   - GPUはピクセルを並列処理するが、水彩の「混色」は直前のDabの結果を次のDabに引き継ぐ「直列依存」のため、GPUの並列処理と根本的に相性が悪い
   - 個人開発のメンテナンス性も崩壊する
   
2. **ストローク・メッシュ（リボン方式）案（却下）**: 
   - テクスチャ（`.sut` / `.abr`）との完全な矛盾
   - 自己交差時の混色バグが解決困難

3. **最終決定**: **CPU（Rust SIMD）で描画・混色、GPUは表示専念**

#### 最終アーキテクチャ

```
[入力スレッド]
  PointerEvent → RawPoint(x, y, pressure, tilt, timestamp)
     ↓ 高頻度サンプリング（120Hz以上）
[補正スレッド（Rust）]
  多段LPFによる平滑化 → Dab配列の生成
     ↓
[ブラシエンジン（CPU SIMD）]
  SAI式混色アルゴリズム on 256x256タイル
  ├── ベクターパス系（シンプルブラシ）→ CPU SDF計算
  └── ラスターDab系（水彩・テクスチャ）→ CPU SIMD混色
     ↓
[タイルキャッシュ（メモリ）]
  変更タイルのみ差分更新
     ↓
[GPU（WebGPU）]
  テクスチャとして転送 → レンダリング（表示のみ）
```

#### デュアル・ブラシエンジン（用途別パイプライン分離）

| ブラシ種別 | エンジン | 説明 |
|-----------|---------|------|
| **ベクターパス系** | CPU SDF | 鉛筆・ペン・マーカー。数学的に入り抜きを計算。「ヘビ柄問題」なし |
| **ラスターDab系** | CPU SIMD + Workgroup Shared Memory | 水彩・厚塗り・テクスチャ系。`.sut` / `.abr` も対象 |

## 9. ストローク・アルゴリズムパイプライン（数理モデル）

「ハードウェアがどう処理するか」は一旦忘れ、ペンの入力から最終的な色が出力されるまでの**「数学的なアルゴリズム（データ変換のプロセス）」**のみに焦点を当てた論理モデル。

### パイプライン全体像

```
[生入力: PointerEvent]
     ↓ Step 1: 入力平滑化アルゴリズム
[平滑化された座標列]
     ↓ Step 2: 動的サンプリング・アルゴリズム（Dab Interpolation）
[均一間隔のDab配列]
     ↓ Step 3: 水彩混色アルゴリズム（Color Blending & Smudge Model）
[最終ピクセル値]
```

### Step 1: 入力平滑化アルゴリズム（Smoothing Algorithm）

SAIのような「吸い付くような描き味」を生み出す手ブレ補正の数学的モデル。

**採用: 多段ローパスフィルタ（Multistage LPF）**

単純な移動平均（過去数点の座標の平均）では「もっさりとした遅延」が発生し、角が丸まりすぎる。

```
計算式: P_out = P_prev + (P_in - P_prev) × α

- α（平滑化係数）: 0.0〜1.0の値。補正が強いほど小さくなる
- 多段処理: 上記の計算をk回繰り返す（k=1〜20）
```

**物理的直感**: 質量バネ・ダンパモデル（Mass-Spring-Damper System）として解釈可能。ゆっくり描くときはバネの力でしっかり追従、速く動かすときは慣性が勝って軌跡が滑らかになる。

**Rust実装方針**:
- 座標データは `f32` で処理
- 120Hz以上の高頻度サンプリングを行い、入力が途切れた際も補間点を出力し続ける「入力予測タイマー」を実装

### Step 2: 動的サンプリング・アルゴリズム（Dab Interpolation）

**採用: 動的Dab密度制御（Dynamic Spacing）**

ブラシのスタンプ（Dab）は、「ペンが進んだ距離」がブラシサイズのN%になるたびに1個押す。Nをパラメータとして調整することで、線の密度を制御する。

### Step 3: 水彩混色アルゴリズム（Color Blending & Smudge Model）

**採用: MyPaint方式「Color Reservoir（絵の具溜まり）」+ SAI式の融合**

#### パラメータ

| パラメータ | 説明 |
|-----------|------|
| `C_fg` | ブラシに設定された色（絵の具） |
| `C_brush` | ブラシが現在「吸い込んでいる」色 |
| `Amount`（絵の具の量） | 0.0〜1.0 |
| `Wetness`（水分量・色伸び） | 0.0〜1.0 |

#### 混色ステップの数理モデル

**①吸い上げ（Pickup）**:
```
C_brush = (C_bg × Wetness) + (C_fg × (1 - Wetness))
```
ペンがキャンバスに触れた瞬間、ブラシの色は前景色と下地を水分量の比率で混ぜた色になる。

**②キャンバスへの定着（Deposit）**:
```
C_new_bg = (C_brush × Amount) + (C_bg × (1 - Amount))
```
混ざったブラシの色を、絵の具の量（筆圧に依存）を使ってキャンバスに定着させる。

**③色の枯渇**:
線を引く（Dabを進める）ごとに、ブラシが吸い込んでいる色は徐々に元の前景色に戻っていくように減衰係数を掛ける。

#### 重要：Premultiplied Alpha（乗算済みアルファ）

透明部分のフチが黒く濁る「Muddy Color」問題を防ぐため、すべてのピクセル演算は乗算済みアルファ形式で行う。これは絶対に妥協してはいけない数学的要件。

```
格納形式: (R×A, G×A, B×A, A)
演算後: 最終的に (R×A / A, G×A / A, B×A / A, A) に戻す
```

### 先行ツールのアルゴリズム比較

| ツール | 方式 | LibreCanvasでの扱い |
|--------|------|-------------------|
| **Krita (libmypaint)** | Color Reservoir + 物理ベース混色。パラメータ多数（数十個） | データ構造を参考にする。ただし全パラメータ実装は過剰 |
| **Photoshop** | ピクセル・スタンピング + デュアルブラシ + 散布（ジッター） | `.abr` 互換のため基本的なスタンプ・散布計算式は将来実装 |
| **Procreate** | モバイル最適化した独自Metal/iOS実装 | 参考にとどめる |
| **SAI** | 独自の軽量混色（詳細非公開）。「吸い付く線」が特徴 | **我々のベンチマーク。数理モデルを再実装** |

### 最強のアルゴリズム構成（LibreCanvas採用）

| レイヤー | 採用アルゴリズム | 理由 |
|---------|----------------|------|
| **入力・補正系** | 多段ローパスフィルタ（EMA×k） | 計算コストほぼゼロ。SAIの「補正なのに遅延を感じない」を再現 |
| **描画系** | Dynamic Spacing（Dab密度動的制御） | シンプルで実装が軽い |
| **混色系** | MyPaint方式 Color Reservoir | 物理的に正しく計算が軽い。将来的なOSS標準互換性を担保 |
| **α計算** | Premultiplied Alpha強制 | Muddy Colorを完全に排除 |

## 10. 却下されたアイデアと教訓（回避すべき罠）

### 罠①: GPUオーバーエンジニアリング（2026/04/11 議論）

- **初期案**: WebGPU Compute Shader「Workgroup Shared Memory」で混色を超並列処理
- **却下理由**: 水彩の混色は「直列依存」のため、GPUの並列処理と根本的に相性が悪い。メンテナンスも崩壊する
- **教訓**: 「新しい技術を使うこと」を目的化しない。**描画・混色はCPU（SIMD）で行い、GPUは表示に専念させる**

### 罠②: アルゴリズムの過度な単純化（2026/04/11 議論）

- **初期案**: 3つの変数の単純な加重平均だけで混色を実装する
- **却下理由**: 透明部分のフチが黒く濁る「Muddy Color」問題が発生する
- **教訓**: **「乗算済みアルファ（Premultiplied Alpha）」と「ピクセル領域の引きずり（Smear）」**は、計算が重くなっても絶対に妥協してはいけない数学的要件

### 罠③: 車輪の再発明（2026/04/11 議論）

- **初期案**: 独自のブラシパラメータ体系を一から定義する
- **却下理由**: 既存のブラシ資産を活かせず、ゼロからの調整に膨大な時間がかかる
- **教訓**: libmypaintのデータ構造など、**既存のOSS標準をベースに、SAIの計算式を組み込む「キメラ戦略」**が最も効率的

### 罠④: ストローク・メッシュ（リボン方式）の採用（2026/04/11 議論）

- **初期案**: ストロークをポリゴンメッシュとして表現し、GPUで描画
- **却下理由**:
  1. 質感（テクスチャ）の完全な喪失と「ヘビ柄問題」（長い線が蛇のようなヘビ柄になる）
  2. `.sut` / `.abr` の読み込みとの完全な矛盾（ラスターベースのブラシが読めない）
  3. 自己交差（Self-Intersection）時の混色バグが解決困難
- **教訓**: ラスターベースのブラシ互換性を保つためには、ラスターDabエンジンが必要

## 12. 詳細技術仕様書

### システムアーキテクチャ（Core-Shell Model）

```
[Core（描画エンジン）]
  ├── 1. 入力層: PointerEvent → RawPoint(x, y, pressure, tilt, timestamp)
  ├── 2. 補正層: 多段LPF → Vec<SmoothPoint>
  ├── 3. Dab生成層: Dynamic Spacing → Vec<Dab>
  ├── 4. 混色層: SAI/MyPaint Hybrid → PixelDiff（タイル単位）
  └── 5. ストレージ層: タイルI/O（zstd圧縮）

[Shell（Tauri / SwiftUI）]
  ├── ファイルシステムアクセス（クラウド同期フォルダへの読み書き）
  ├── 自動バックアップ（.bak生成）
  └── OS統合（ショートカット、ファイルアソシエーション）
```

### 1. 入力・補正系の仕様

**① 入力層（Input handling）**

ブラウザの `PointerEvent` からデータを取得し、Rustへ渡す構造体:

```rust
struct RawPoint {
    x: f32,
    y: f32,
    pressure: f32,    // 0.0〜1.0
    tilt_x: f32,      // -90〜90度
    tilt_y: f32,
    timestamp: u64,   // ミリ秒
}
```

**② 補正層（Smoothing）**

多段LPFを適用し、補間されたDab（スタンプ）の配列を生成:

```
アルゴリズム: 指数移動平均（EMA）の k 回繰り返し
  P_out = P_prev + (P_in - P_prev) × α
出力: Vec<Dab>（座標、筆圧、角度が計算済みの配列）
```

**③ 描画層（Rendering - SIMD Kernel）**

混色の計算式（最重要）:

```
ターゲット: 32bit RGBA（Premultiplied Alpha）
SIMD命令: f32x4 を使い、4つのチャネル（R, G, B, A）を同時に演算
```

### 2. タイルベース・メモリ管理

キャンバス全体を保持せず、**Sparse Matrix（疎行列）**的な管理を行う。

| 項目 | 仕様 |
|------|------|
| **Tile Unit** | 256×256 ピクセル |
| **Storage** | 変更があったタイルのみを `HashMap<u64, Tile>` で保持 |
| **Key** | タイルの2次元インデックスを1次元にハッシュ化したもの |
| **Dirty Rect管理** | ストロークが通過したタイルのインデックスをスタックし、描画フレームごとにそのタイルだけを更新対象としてマーク |
| **読み飛ばし** | 完全な透明レイヤー、または変更がないタイルは計算から完全に除外 |

### 3. 同期・永続化系（BYOS）

**ファイル構造（分散型 `.lcv` 形式）**:

```
artwork.lcv/
├── meta.json               ← キャンバス情報（サイズ、DPI、レイヤー一覧）
├── tiles/
│   ├── layer_0/
│   │   ├── 0_0.bin        ← 256x256タイル（zstd圧縮済みRGBA）
│   │   └── ...
│   └── layer_1/
│       └── ...
└── thumbnail.png
```

**bk（バックアップ）戦略**:
- `journal.log` は各デバイスの `AppData` / `Application Support` 内にローカル保存
- クラウド側のファイルが破損しても、このログからストロークを再実行（Playback）することで絵を100%復元可能

### 4. 開発ログ：却下されたアイデアと回避策

| | 初期案 | 回避策（最適解） | 理由 |
|--|--------|----------------|------|
| **描画エンジン** | WebGPU Compute Shader | Rust SIMD（CPU） | 混色の直列依存性とメンテナンス性を重視 |
| **ファイル保存** | 巨大な単一バイナリ | タイル別個別ファイル | クラウドの差分同期と相性が良い |
| **ブラシ形式** | 独自パラメータ体系 | libmypaint互換ベース | 既存ブラシ資産の活用と実装効率 |
| **混色計算** | 単純加重平均 | Premultiplied Alpha + Reservoir | Muddy Color問題の回避 |
| **同期** | 独自サーバー / CRDT | OSクラウドに丸投げ | 開発・維持コストゼロ |
| **ストローク表現** | リボン方式（メッシュ） | ラスターDab | テクスチャ互換性と混色の正確さ |

## 13. PoC実装ガイド

### 開発環境のセットアップ（Toolchain）

```bash
# Rust（Nightly推奨：std::simd を使用するため）
rustup toolchain install nightly
rustup default nightly

# WebAssembly ターゲットの追加
rustup target add wasm32-unknown-unknown

# wasm-pack（RustをWasmにビルド）
cargo install wasm-pack

# プロジェクト構成
# core/  → Rustによる描画エンジン
# web/   → TypeScript + Vite
# app/   → Tauri
```

### 主要クレート（Dependencies）

```toml
[dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"
zstd = "0.13"
wasm-bindgen = "0.2"

[target.'cfg(not(target_arch = "wasm32"))'.dependencies]
rayon = "1"   # デスクトップ版のみ（Wasm環境では使用不可）
```

### 実装コア：描画パイプラインのデータフロー

```
[TypeScript / UI]
  PointerEvent → RawPoint(x, y, pressure, tilt, timestamp)
     ↓ wasm-bindgen経由でRustへ渡す
[Rust Core]
  ① 多段LPF（EMA × k）による平滑化
  ② Dynamic Spacing によるDab配列生成 → Vec<Dab>
  ③ SIMD混色カーネル（SAI/MyPaint式）→ タイルのPixelDiff
     ↓
[TypeScript / Canvas]
  変更タイルのみをImageDataとしてWebGPU/Canvasに転送 → 表示
```

### PoCで検証すべき成功指標（KPI）

| KPI | 目標値 | 測定方法 |
|-----|--------|---------|
| **入力遅延** | ペンダウンから描画まで16ms以内（60fps相当） | HighResolutionTimestamp で計測 |
| **ストロークFPS** | 4Kキャンバス、ペン先100px、60fps以上維持 | requestAnimationFrame ループで計測 |
| **メモリ使用量** | 4Kキャンバス（全タイル使用時）で500MB以下 | Chrome DevTools Memory |
| **保存速度** | 差分タイル保存（1ストローク分）が100ms以内 | File API のタイムスタンプで計測 |
| **主観的描き心地** | 開発者本人が「SAIと同等以上」と感じること | 主観評価 |

### 回避すべき既知のトラブル（Tips）

| トラブル | 回避策 |
|---------|--------|
| Wasm内で `rayon` を使うと動かない | デスクトップ版のみ `cfg` で条件分岐 |
| `std::simd` はNightly限定 | `toolchain.toml` で `nightly` を固定 |
| Premultiplied AlphaをWebGPUで表示すると色がおかしい | `GPUBlendFactor::One` / `GPUBlendFactor::OneMinusSrcAlpha` を使う |
| タイルまたぎのストロークで継ぎ目が見える | Dab生成時にタイル境界を超えて±1タイル分マージンを持たせる |
