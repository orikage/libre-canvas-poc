# LibreCanvas PoC 概要ドキュメント

> 次世代オープンソース・ペイントツール「LibreCanvas」（仮称）  
> 軽量・同期・訴訟対策を軸とした企画

---

## 1. プロジェクト概要

### コンセプト

サブスクリプションモデルと多機能化による肥大化に対するアンチテーゼとして、**極限まで軽量化された完全無料のオープンソース・ペイントツール**を開発する。

- 中央集権的なサーバーや通信機能を一切持たない
- データ管理と同期はOSやユーザー既存のクラウドサービスに委ねる
- 開発・維持コストの永続的なゼロ化とユーザーの完全なデータ主権を実現

### ベンチマーク

**2015年頃のデジタルペイントツール水準**（初期クリスタ、SAI、Photoshop CS6/CC 2015等）

この時代のツールは描画の基本性能がすでに完成されており、「これさえあれば描ける」コア機能が凝縮されている。現在の肥大化した機能を削ぎ落とし、**軽量・高速・高互換**な設計を目指す。

---

## 2. 解決すべき課題（Problem）

| 課題 | 詳細 |
|------|------|
| **過剰な多機能化・重量化** | 3Dモデル連携・スマートシェイプ等の追加により起動時間・メモリが増大。「ちょっと描きたい」という初期衝動を阻害 |
| **サブスクリプションによるロックイン** | たまにしか使わないユーザーへの継続課金負担。解約後は過去データの編集すら困難 |
| **データ主権の喪失** | 複数デバイス同期が各社の独自クラウドに依存。容量制限・サービス終了リスクと常に隣り合わせ |
| **商業的ノイズ** | 全画面広告、ブラシのロック解除要求、行動トラッキング等による思考の分断 |

### ユーザーの声（PoC発端）

> 「ちょろっと書きたいだけなのにサブスクで払いたくない。PCとiPadでデータを同期しておきたい。」

---

## 3. ターゲットユーザー

- 高度な機能は不要で、純粋に「描く」ことだけをサクッと行いたい**ライトユーザー・ホビー層**
- 自分のデータは自分で管理したいプライバシー意識の高いユーザー
- 複数デバイス（PC + iPad等）をまたいで作業するユーザー

---

## 4. コアバリュー

### ユーザー目線

- **サブスクからの解放**: ずっと無料、課金圧力が一切ない
- **「描きたい時にすぐ描ける」**: 重厚な起動プロセスを排除した圧倒的な軽量性
- **シームレスな体験**: PC（Win/Mac/Linux）で描きかけのラフをiPadからそのまま続けられる
- **完全なノイズレス**: 広告・トラッキング・ポップアップなし

### 運営・開発目線

- **インフラコストゼロ**: 中央サーバー不要。OSS維持コストを最小化
- **持続可能な開発モデル**: GitHub Sponsors等のドネーションとコミュニティ主導
- **法的リスクの完全排除**: クリーンルーム設計と機能のモジュール化

## 5. アーキテクチャ設計

### 基本方針

```
アプリ本体 = 通信機能を持たない純粋な「超軽量ローカル・ペイントツール」
```

### 技術スタック

| レイヤー | 採用技術 | 理由 |
|---------|---------|------|
| **コア描画エンジン** | Rust + WebGPU / WASM | 高速・低遅延・クロスプラットフォーム |
| **デスクトップ（Win/Mac/Linux）** | Tauri | Electronより軽量、Rustバックエンドと親和性が高い |
| **Web版（ベースライン）** | ブラウザ上で動作する Canvas / WebGPU | アーキテクチャをクリーンに保ち、OSSとしての寿命を延ばす |
| **iPad（将来）** | Tauri Mobile または SwiftUI ラッパー | OSネイティブのファイルアプリと深く連携 |
| **プラグイン** | WASM（WebAssembly）モジュール | ブラシインポーター等をサードパーティが追加可能 |

### データフロー

```
[ペイント操作]
     ↓
[LibreCanvas コア（通信なし）]
     ↓ 保存
[ユーザーのクラウド同期フォルダ]  ← OneDrive / iCloud Drive 等
     ↓                              ユーザーが選択
[PC / iPad から同じファイルを開く]
```

**同期はユーザー自身のクラウドサービスに委ねる（BYOS: Bring Your Own Storage）**

### バックアップ戦略

```
同期フォルダ: artwork.lcv（常に最新の保存ファイル）
ローカル:     artwork_20260411_1650.bak（各デバイスにローテーション自動生成）
```

- クラウドの同期ミスでファイルが壊れても、各デバイスのローカルバックアップから復旧可能
- バックアップは各デバイスに置き、同期対象には含めない

## 6. ファイル形式・ブラシ互換性

### 独自フォーマット

- **`.lcv`（LibreCanvas Vector）**: キャンバスファイル
- **`.lcb`（LibreCanvas Brush）**: JSON/YAMLベースのオープンなブラシ形式
  - 格納内容: 形状画像（PNG）、テクスチャ、筆圧感度曲線、混色アルゴリズムのパラメータ

### 他形式との互換性（法的リスク回避設計）

| アプローチ | 詳細 |
|-----------|------|
| **コア本体はクリーン** | `.sut`（クリスタ）や`.abr`（Photoshop）のインポート機能は本体に含まない |
| **プラグイン化** | ブラシ読み込み機能をWASMモジュールとして外部化。コミュニティが自由に開発・配布 |
| **ユーザー主導** | 開発者が公式に配布するのではなく、「ユーザーが作ったインポーターを後から追加できる仕組み」だけを提供 |

**UXフロー**:
1. ユーザーが `.abr` や `.sut` ファイルをD&D
2. 対応インポーターがあれば即座に `.lcb` に変換してパレットに追加
3. なければ「〇〇インポーターが必要です。コミュニティサイトで探しますか？」と表示

## 7. 核心描画機能要件（SAIレベルを目標）

### 最重点投資の3要件

#### ① 「吸い付く線」のアルゴリズム（Stroke Smoothing）

SAIの「補正されているのに遅延を感じさせない、滑らかで尖った抜き」の再現。

- Ramer-Douglas-Peuckerアルゴリズムの応用
- Catmull-Romスプライン / ベジェ曲線による補間
- RustとWebGPUの演算能力でチューニング

#### ② リアルタイムな「混色」と「色伸び」（Color Blending）

SAI水彩筆のような下地の色を拾いながら滑らかに混ざる表現。

- 単純なアルファブレンドではなく顔料の混色モデル（Kubelka-Munk理論の簡易版）
- 軽量・高速なシェーダー実装

#### ③ 高密度・低ノイズUI（High-Density, Low-Noise UI）

SAIのように視線移動が最小限で済む、削ぎ落とされたUI。

- モダンなフラットデザインとSAIの情報密度を融合
- タブレット（iPad）操作を前提とした大きめのタッチターゲット

### その他コア機能

| 機能 | 詳細 |
|------|------|
| 低遅延・高FPS描画 | 4Kキャンバスでもブラシストロークが遅延なく追従。WebGPU/WASMハードウェア加速 |
| 筆圧・傾き検知 | 基本的な筆圧感度曲線のカスタマイズ |
| 手ブレ補正 | 2015年当時の標準レベル以上 |
| レイヤー管理 | 通常・乗算・スクリーン等の基本合成モード |
| 取り消し/やり直し | 十分な履歴数 |

## 9. 訴訟対策（リーガル・リスクマネジメント）

### 避けるべきこと

- `.sut` 等の独自フォーマットをリバースエンジニアリングしてコアに組み込む
- 「クリスタ互換」「クリスタの代替」を前面に出したマーケティング
- CelSYS社等の商標・ロゴの無断使用

### 実践すべき対策

| 対策 | 内容 |
|------|------|
| **クリーンルーム設計** | 既存商用ツールのソースコードを参照せず、仕様のみを公開情報から実装 |
| **機能のモジュール化** | 互換性機能はコアから分離し、コミュニティプラグインとして提供 |
| **オープンライセンス** | MIT または Apache 2.0 ライセンスを採用。特許リスクに注意 |
| **コントリビューションガイド** | 著作権侵害リスクのあるコード提出を防ぐDCO（Developer Certificate of Origin）を導入 |
| **記録の保持** | 設計決定の根拠と情報源を記録し、独自開発であることを証明できる状態を維持 |

## 12. 技術的リスクと対策

| リスク | 対策 |
|--------|------|
| WebGPUのブラウザ対応状況 | WebGL2フォールバックを用意。デスクトップ版はネイティブWGPUを使用 |
| iPad版の実現難易度 | Phase 1ではWebApp（PWA）として提供し、Phase 2でネイティブ化 |
| クラウド競合ファイル | ローカルバックアップで対処。将来的にConflict Resolutionロジックを追加 |
| OSS維持継続性 | モノレポ構成・明確なコントリビューションガイドで属人化を防ぐ |
